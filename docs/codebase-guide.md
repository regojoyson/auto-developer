# Codebase Guide

A map of every file and folder in the project — what it does, when it's touched, and how the pieces fit together.

---

## Top-Level Structure

```
auto-pilot/
├── agents/              # Agent system prompts (Markdown)
├── src/                 # Python server source code
├── installer/           # TUI setup wizard
├── mcp_servers/         # Built-in MCP servers (GitLab, GitHub)
├── scripts/             # Helper shell/Python scripts
├── console/             # Optional React web console
├── docs/                # All documentation
├── config.yaml          # Main user config (generated by setup.sh)
├── .env                 # Secrets only (tokens)
├── setup.sh             # Entry point for interactive setup
├── start.sh             # Entry point for running the pipeline
└── stop.sh              # Stops the pipeline
```

## `agents/` — AI Agent Prompts

Markdown files that define what each AI agent does. The `runner.py` prepends an autonomy preamble and passes these as system prompts to the CLI.

| File | Purpose |
|------|---------|
| `orchestrator.md` | The main pipeline agent. Split into 5 actions: `analyze`, `plan`, `implement`, `rework`, `merge-approved`, `rework-limit-exceeded`. Python server invokes it once per phase with a different action. |
| `brainstorm.md` | Invoked by orchestrator during the `plan` action. Reads TICKET.md, explores the codebase, writes PLAN.md. |
| `developer.md` | Invoked by orchestrator during `implement` and `rework` actions. Reads PLAN.md (or FEEDBACK.md) and writes actual code, commits, pushes. |
| `feedback-parser.md` | Invoked during rework phase. Reads review comments on the MR and writes FEEDBACK.md for the developer agent to act on. |
| `RULES.md` | Global rules applied to all agents — no-interaction mode, skill-ignoring, commit message format, etc. |

## `src/` — Python Server

The FastAPI pipeline controller.

### `src/server.py`
Application entry point. Registers routes, starts uvicorn.
```
GET  /health
POST /webhooks/issue-tracker
POST /webhooks/git
POST /api/trigger
GET  /api/status
GET  /api/status/{issueKey}
GET  /api/status/{issueKey}/logs
DELETE /api/status/{issueKey}
```

### `src/config.py`
Reads and parses `config.yaml` into a cached Python dict. Splits `issueTracker.type` (e.g. `jira-api`) into `platform` + `api_mode` for internal use. Auto-loads on first import so `from src.config import config` just works.

### `src/state/manager.py`
Pipeline state machine. Each pipeline's state is stored as JSON in `.pipeline-state/{branch}.json`. Provides:
- `create_state(branch, issue_key)` — initial state = `analyzing`
- `get_state(branch)` — read current state
- `transition_state(branch, new_state, error=None)` — validated state transition
- `record_phase_start(branch, phase, agent)` / `record_phase_end(branch, exit_code, result)` — tracks phase timing in a `phases[]` array
- `update_artifacts(branch, **kwargs)` — stores MR URL, PR ID, etc.
- `list_active_states()`, `delete_state()`, `is_rework_limit_exceeded()`

Valid state transitions:
```
analyzing → planning/failed
planning → developing/failed
developing → awaiting-review/failed
awaiting-review → reworking/merged
reworking → awaiting-review/failed
```

### `src/executor/`
The engine that runs agents and drives the pipeline.

| File | Purpose |
|------|---------|
| `runner.py` | Spawns the AI CLI as a subprocess, streams stdout/stderr to output handlers line by line. Prepends `AUTONOMY_PREAMBLE` + `SKILLS_DISABLED_PREAMBLE` (if configured) + `NO_SLACK_PREAMBLE` (if notifications disabled) to every agent input. |
| `pipeline.py` | **The core controller.** `run_pipeline_phases()` drives analyze → plan → implement. `run_rework_phases()` drives rework. Handles state transitions, Jira API calls (in api mode), error handling, best-effort Slack notifications. Phase-by-phase orchestration. |

### `src/routes/`
FastAPI route handlers.

| File | Endpoint | What it does |
|------|----------|-------------|
| `issue_tracker.py` | `POST /webhooks/issue-tracker` | Receives Jira/GitHub Issues webhooks. Parses event, creates state, spawns `run_pipeline_phases()` in a background thread. |
| `trigger.py` | `POST /api/trigger` | Manual trigger endpoint (no webhook needed). Same flow as issue_tracker. |
| `git_provider.py` | `POST /webhooks/git` | Receives GitLab/GitHub MR webhooks. Handles `approved` (→ merged), `push` (log human edits), `comment` (→ spawns `run_rework_phases()`). |
| `status.py` | `GET /api/status`, `GET /api/status/{k}`, `GET /api/status/{k}/logs`, `DELETE /api/status/{k}` | Read/delete pipeline state, fetch real-time logs. |

### `src/providers/`
Pluggable adapters — the extensibility layer. Each category has a base class + concrete implementations.

#### `src/providers/base.py`
Abstract base classes:
- `IssueTrackerBase` — `parse_webhook`, `read_issue`, `transition_issue`, `add_comment`
- `GitProviderBase` — `parse_webhook`, `create_api` (returns API client with 8 methods)
- `CliAdapterBase` — `build_args`, `parse_output`, `build_env`
- `NotificationBase` — `send`
- `OutputHandlerBase` — `on_start`, `on_output`, `on_finish`, `get_output`, `delete_logs`

#### `src/providers/issue_tracker.py` (factory)
Returns the configured issue tracker adapter based on `config.issue_tracker.platform`. Also exposes `is_api_mode()` helper.

#### `src/providers/trackers/`
Issue tracker implementations:
- `jira.py` — Parses Jira webhooks. REST API methods (`read_issue`, `transition_issue`, `add_comment`) with v3 → v2 fallback and ADF text extraction.
- `github_issues.py` — Parses GitHub Issues webhooks. REST API methods.

#### `src/providers/git_provider.py` (factory)
Returns configured git provider adapter.

#### `src/providers/git/`
- `gitlab.py` — Parses GitLab MR webhooks. `create_api()` returns a client with `create_branch`, `commit_files`, `create_pr`, `list_pr_comments`, etc.
- `github.py` — Same for GitHub.

#### `src/providers/cli_adapter.py` (factory)
Returns configured AI CLI adapter.

#### `src/providers/cli/`
- `claude_code.py` — Claude Code CLI args/output parsing
- `codex.py` — Codex CLI
- `gemini.py` — Gemini CLI

#### `src/providers/notification.py` (factory)
Returns the configured notification adapter, or `None` if no `notification:` section in config.

#### `src/providers/notifications/`
- `slack.py` — Currently a stub that logs messages. Real Slack integration would go here.

#### `src/providers/output_handler.py` (registry)
Fans out agent output to all enabled handlers. `get_output_handlers()` is cached.

#### `src/providers/output/`
- `file_handler.py` — Writes agent output to `logs/agents/{issueKey}-{agent}.log` with line buffering. Auto-opens files on first write.
- `memory_handler.py` — Keeps output in in-memory buffers (`defaultdict`) for API access. Cleared via `delete_logs`.

### `src/repos/`
Repository management.

| File | Purpose |
|------|---------|
| `resolver.py` | Resolves target repo path based on `repo.mode` (dir / parentDir / clone). `prepare_repo()` does stash + checkout base + reset to origin. `get_base_branch()` returns configured base. |
| `git_remote.py` | Parses git remote URLs to extract owner/repo info. Used by git provider MCP servers. |

## `mcp_servers/` — Built-in MCP Servers

Custom MCP servers that `start.sh` auto-configures in the CLI's settings.json. Agents use these to interact with GitLab/GitHub via API.

| File | Purpose |
|------|---------|
| `gitlab_server.py` | Exposes 8 tools: `create_branch`, `commit_files`, `create_merge_request`, `get_merge_request`, `update_merge_request`, `list_mr_comments`, `post_mr_comment`, `get_file`. |
| `github_server.py` | Same 8 tools for GitHub. |

Note: Issue tracker MCP servers (Jira, Slack) are **not** built-in — user configures those in their CLI separately when using `*-mcp` mode.

## `console/` — Web UI (Optional)

React + Vite single-page app. Polls the API every 3 seconds for live updates.

### `console/src/App.jsx`
Top-level routing. 3 routes:
- `/` — `PipelineList`
- `/trigger` — `TriggerForm`
- `/pipeline/:issueKey` — `PipelineDetail` (combined status + logs view)

### `console/src/api.js`
Thin wrapper around `fetch` for the server API. Methods: `listPipelines`, `getPipeline`, `getLogs`, `trigger`, `cancel`.

### `console/src/hooks/usePolling.js`
3-second polling hook used by list and detail pages.

### `console/src/components/`
| File | Purpose |
|------|---------|
| `Sidebar.jsx` | Left nav with links to Pipelines and Trigger Pipeline. Live indicator dot. |
| `PipelineList.jsx` | Main page — table of all active pipelines with state badges. Click "Open" to view detail. |
| `PipelineDetail.jsx` | **The combined view** — pipeline progress (5-step bar with per-phase timing), collapsible details card, live agent logs with search/filter/copy. |
| `TriggerForm.jsx` | Manual trigger form — submits to `POST /api/trigger`. |
| `ToastContext.jsx` | Toast notification provider used across the app. |

### `console/src/styles.css`
Tailwind + custom CSS for badges (`.badge-analyzing`, `.badge-failed`, etc.), log viewer styling, pipeline table.

## Data Flow: From Ticket to PR

```
1. Jira ticket moves to "Ready for Development"
   └── webhook → POST /webhooks/issue-tracker
       └── src/routes/issue_tracker.py
           └── adapter.parse_webhook() (src/providers/trackers/jira.py)
           └── create_state() (src/state/manager.py)
           └── threading.Thread(target=run_pipeline_phases, ...)

2. run_pipeline_phases() (src/executor/pipeline.py)
   ├── _prepare_repo_for_branch() — stash, checkout base, pull
   ├── adapter.transition_issue() — Ready for Dev → Development [api mode only]
   ├── adapter.read_issue() — [api mode only, passes ticketData to agent]
   │
   ├── Phase 1 — Analyze
   │   └── _run_phase("orchestrator", "orchestrator:analyze")
   │       └── run_agent() (src/executor/runner.py)
   │           └── subprocess.Popen(claude, ...)
   │               └── reads agents/orchestrator.md (action=analyze)
   │               └── creates feature branch via Git MCP
   │               └── writes TICKET.md, commits via Git MCP
   │               └── posts analysis comment on Jira
   │               └── outputs __PIPELINE_RESULT__:{"blocked":false}
   │   └── record_phase_end(success)
   │
   ├── Phase 2 — Plan
   │   └── _run_phase("orchestrator", "orchestrator:plan")
   │       └── orchestrator invokes brainstorm agent
   │           └── brainstorm writes PLAN.md
   │       └── orchestrator posts plan comment on Jira
   │
   ├── Phase 3 — Implement
   │   └── _prepare_repo_for_branch(feature_branch=branch) — checkout
   │   └── _run_phase("orchestrator", "orchestrator:implement")
   │       └── orchestrator invokes developer agent
   │           └── developer writes code, commits locally
   │       └── orchestrator runs git push origin <branch>
   │       └── orchestrator creates MR via Git MCP
   │
   └── transition_state("awaiting-review")
       └── adapter.transition_issue() — Development → Done [api mode only]
       └── _try_notify_slack() — [only if notification enabled]

3. Human reviews the MR on GitLab/GitHub
   ├── approves → webhook → POST /webhooks/git (event=approved)
   │   └── transition_state("merged")
   │
   └── comments → webhook → POST /webhooks/git (event=comment)
       └── threading.Thread(target=run_rework_phases, ...)
           ├── feedback-parser agent reads MR comments → FEEDBACK.md
           └── orchestrator invokes developer (rework mode) → applies fixes
           └── transition_state("awaiting-review")
```

## Which Config Controls What?

| Config Field | Affects |
|--------------|---------|
| `repo.mode` | `src/repos/resolver.py` — how to find the target repo |
| `repo.baseBranch` | `src/repos/resolver.py`, `pipeline.py` — branch to create features from |
| `issueTracker.type` | `config.py` splits into `platform` + `api_mode`; factory in `issue_tracker.py` picks adapter; `pipeline.py` behavior (api reads ticket, mcp doesn't) |
| `issueTracker.*Status` | Passed to agent via `statuses` input; Python uses in `transition_issue()` calls |
| `gitProvider.type` | Factory in `git_provider.py` picks adapter; `start.sh` generates matching MCP settings.json |
| `cliAdapter.type` | Factory in `cli_adapter.py` picks how to invoke the CLI |
| `notification` (section) | If missing → `get_notification()` returns None → Python never calls Slack → runner adds NO_SLACK_PREAMBLE to agent prompt |
| `pipeline.maxReworkIterations` | Checked in `routes/git_provider.py` — escalates after N reworks |
| `pipeline.allowCliSkills` | `runner.py` — if false, prepends SKILLS_DISABLED_PREAMBLE to agent input |
| `pipeline.outputHandlers` | `output_handler.py` registry — which handlers to load (file, memory, or both) |
| `pipeline.port` | `start.sh` — uvicorn bind port |
| `pipeline.agentTimeout` | `runner.py` — agent subprocess timeout |

## Where to Make Common Changes

| I want to... | Edit |
|--------------|------|
| Add a new issue tracker | `src/providers/trackers/newtracker.py` + register in `issue_tracker.py` factory + add to `choices.py` |
| Add a new git provider | `src/providers/git/newgit.py` + register + add MCP server in `mcp_servers/` |
| Add a new AI CLI | `src/providers/cli/newcli.py` + register + add to `choices.py` |
| Change agent behavior | Edit `.md` file in `agents/` |
| Add a new pipeline phase | Add to `VALID_TRANSITIONS` in `state/manager.py`, add phase in `pipeline.py`, add action in `orchestrator.md` |
| Change what the TUI asks | `installer/setup.py` + `installer/choices.py` |
| Change state file structure | `src/state/manager.py` (and update dashboard if shown) |
| Add an API endpoint | New file in `src/routes/` + register in `server.py` |
| Change dashboard appearance | `console/src/components/*.jsx` + `styles.css` |
| Add a log handler (websocket, webhook, etc.) | New file in `src/providers/output/` extending `OutputHandlerBase` + register in `output_handler.py` |
