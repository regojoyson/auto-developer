## `agents/` вЂ” AI Agent Prompts

Markdown files that define what each AI agent does. The `runner.py` prepends an autonomy preamble and passes these as system prompts to the CLI.

| File | Purpose |
|------|---------|
| `analyze.md` | Invoked during analyze phase. Reads ticket data from issue tracker, explores the codebase, writes TICKET.md. |
| `plan.md` | Invoked during plan phase. Reads TICKET.md, explores the codebase, writes PLAN.md. |
| `implement.md` | Invoked during implement phase. Reads PLAN.md and writes actual code, commits locally. |
| `rework.md` | Invoked during rework phase. Reads PLAN.md and FEEDBACK.md, applies fixes, commits locally. |
| `feedback-parser.md` | Invoked during rework phase. Reads review comments on the MR and writes FEEDBACK.md for the rework agent to act on. |
| `RULES.md` | Global rules applied to all agents вЂ” no-interaction mode, skill-ignoring, commit message format, etc. |

## `src/` вЂ” Python Server

The FastAPI pipeline controller.

### `src/server.py`
Application entry point. Registers routes, starts uvicorn.
```
GET  /health
POST /webhooks/issue-tracker
POST /webhooks/git
POST /api/trigger
GET  /api/status
GET  /api/status/{issueKey}
GET  /api/status/{issueKey}/logs
DELETE /api/status/{issueKey}
```

### `src/config.py`
Reads and parses `config.yaml` into a cached Python dict. Splits `issueTracker.type` (e.g. `jira-api`) into `platform` + `api_mode` for internal use. Auto-loads on first import so `from src.config import config` just works.

### `src/state/manager.py`
Pipeline state machine. Each pipeline's state is stored as JSON in `.pipeline-state/{branch}.json`. Provides:
- `create_state(branch, issue_key)` вЂ” initial state = `analyzing`
- `get_state(branch)` вЂ” read current state
- `transition_state(branch, new_state, error=None)` вЂ” validated state transition
- `record_phase_start(branch, phase, agent)` / `record_phase_end(branch, exit_code, result)` вЂ” tracks phase timing in a `phases[]` array
- `update_artifacts(branch, **kwargs)` вЂ” stores MR URL, PR ID, etc.
- `list_active_states()`, `delete_state()`, `is_rework_limit_exceeded()`

Valid state transitions:
```
analyzing в†’ planning/failed
planning в†’ developing/failed
developing в†’ awaiting-review/failed
awaiting-review в†’ reworking/merged
reworking в†’ awaiting-review/failed
```

### `src/executor/`
The engine that runs agents and drives the pipeline.

| File | Purpose |
|------|---------|
| `runner.py` | Spawns the AI CLI as a subprocess, streams stdout/stderr to output handlers line by line. Applies `PhaseScope` restrictions (via `--allowed-tools` / `--disallowed-tools` / `--mcp-config` flags). Prepends `AUTONOMY_PREAMBLE` + `SKILLS_DISABLED_PREAMBLE` (if configured) + `NO_SLACK_PREAMBLE` (if notifications disabled) to every agent input. |
| `pipeline.py` | **The core controller.** `run_pipeline_phases()` invokes analyze в†’ plan в†’ implement phase agents sequentially. `run_rework_phases()` invokes rework phase agent. Handles state transitions, issue tracker API calls (in api mode), error handling, best-effort Slack notifications. Delegates remote git operations to `pipeline_git.py`. |
| `pipeline_git.py` | **Python-driven remote git operations.** Implements `create_feature_branch()`, `commit_via_api()`, `push_to_remote()`, and `create_mr()`. Called from `pipeline.py` after each phase completes. |
| `phase_scope.py` | Per-phase tool sandbox. Defines a `PhaseScope` dataclass with per-phase allowlists for issue tracker operations, git operations, MCP tools, and CLI skills. Used by `runner.py` to restrict agent capabilities. |

### `src/routes/`
FastAPI route handlers.

| File | Endpoint | What it does |
|------|----------|-------------|
| `issue_tracker.py` | `POST /webhooks/issue-tracker` | Receives Jira/GitHub Issues webhooks. Parses event, creates state, spawns `run_pipeline_phases()` in a background thread. |
| `trigger.py` | `POST /api/trigger` | Manual trigger endpoint (no webhook needed). Same flow as issue_tracker. |
| `git_provider.py` | `POST /webhooks/git` | Receives GitLab/GitHub MR webhooks. Handles `approved` (в†’ merged), `push` (log human edits), `comment` (в†’ spawns `run_rework_phases()`). |
| `status.py` | `GET /api/status`, `GET /api/status/{k}`, `GET /api/status/{k}/logs`, `DELETE /api/status/{k}` | Read/delete pipeline state, fetch real-time logs. |

### `src/providers/`
Pluggable adapters вЂ” the extensibility layer. Each category has a base class + concrete implementations.

#### `src/providers/base.py`
Abstract base classes:
- `IssueTrackerBase` вЂ” `parse_webhook`, `read_issue`, `transition_issue`, `add_comment`
- `GitProviderBase` вЂ” `parse_webhook`, `create_api` (returns API client with 8 methods)
- `CliAdapterBase` вЂ” `build_args`, `parse_output`, `build_env`
- `NotificationBase` вЂ” `send`
- `OutputHandlerBase` вЂ” `on_start`, `on_output`, `on_finish`, `get_output`, `delete_logs`

#### `src/providers/issue_tracker.py` (factory)
Returns the configured issue tracker adapter based on `config.issue_tracker.platform`. Also exposes `is_api_mode()` helper.

#### `src/providers/trackers/`
Issue tracker implementations:
- `jira.py` вЂ” Parses Jira webhooks. REST API methods (`read_issue`, `transition_issue`, `add_comment`) with v3 в†’ v2 fallback and ADF text extraction.
- `github_issues.py` вЂ” Parses GitHub Issues webhooks. REST API methods.

#### `src/providers/git_provider.py` (factory)
Returns configured git provider adapter.

#### `src/providers/git/`
- `gitlab.py` вЂ” Parses GitLab MR webhooks. `create_api()` returns a client with `create_branch`, `commit_files`, `create_pr`, `list_pr_comments`, etc.
- `github.py` вЂ” Same for GitHub.

#### `src/providers/cli_adapter.py` (factory)
Returns configured AI CLI adapter.

#### `src/providers/cli/`
- `claude_code.py` вЂ” Claude Code CLI args/output parsing
- `codex.py` вЂ” Codex CLI
- `gemini.py` вЂ” Gemini CLI

#### `src/providers/notification.py` (factory)
Returns the configured notification adapter, or `None` if no `notification:` section in config.

#### `src/providers/notifications/`
- `slack.py` вЂ” Currently a stub that logs messages. Real Slack integration would go here.

#### `src/providers/output_handler.py` (registry)
Fans out agent output to all enabled handlers. `get_output_handlers()` is cached.

#### `src/providers/output/`
- `file_handler.py` вЂ” Writes agent output to `logs/agents/{issueKey}-{agent}.log` with line buffering. Auto-opens files on first write.
- `memory_handler.py` вЂ” Keeps output in in-memory buffers (`defaultdict`) for API access. Cleared via `delete_logs`.

### `src/repos/`
Repository management.

| File | Purpose |
|------|---------|
| `resolver.py` | Resolves target repo path based on `repo.mode` (dir / parentDir / clone). `prepare_repo()` does stash + checkout base + reset to origin. `get_base_branch()` returns configured base. |
| `git_remote.py` | Parses git remote URLs to extract owner/repo info. Used by git provider MCP servers. |

## `mcp_servers/` вЂ” Built-in MCP Servers

Custom MCP servers that `start.sh` auto-configures in the CLI's settings.json. Agents use these to interact with GitLab/GitHub via API.

| File | Purpose |
|------|---------|
| `gitlab_server.py` | Exposes 8 tools: `create_branch`, `commit_files`, `create_merge_request`, `get_merge_request`, `update_merge_request`, `list_mr_comments`, `post_mr_comment`, `get_file`. |
| `github_server.py` | Same 8 tools for GitHub. |

Note: Issue tracker MCP servers (Jira, Slack) are **not** built-in вЂ” user configures those in their CLI separately when using `*-mcp` mode.

## `console/` вЂ” Web UI (Optional)

React + Vite single-page app. Polls the API every 3 seconds for live updates.

### `console/src/App.jsx`
Top-level routing. 3 routes:
- `/` вЂ” `PipelineList`
- `/trigger` вЂ” `TriggerForm`
- `/pipeline/:issueKey` вЂ” `PipelineDetail` (combined status + logs view)

### `console/src/api.js`
Thin wrapper around `fetch` for the server API. Methods: `listPipelines`, `getPipeline`, `getLogs`, `trigger`, `cancel`.

### `console/src/hooks/usePolling.js`
3-second polling hook used by list and detail pages.

### `console/src/components/`
| File | Purpose |
|------|---------|
| `Sidebar.jsx` | Left nav with links to Pipelines and Trigger Pipeline. Live indicator dot. |
| `PipelineList.jsx` | Main page вЂ” table of all active pipelines with state badges. Click "Open" to view detail. |
| `PipelineDetail.jsx` | **The combined view** вЂ” pipeline progress (5-step bar with per-phase timing), collapsible details card, live agent logs with search/filter/copy. |
| `TriggerForm.jsx` | Manual trigger form вЂ” submits to `POST /api/trigger`. |

### `console/src/styles.css`

## Data Flow: From Ticket to PR

```
1. Jira ticket moves to "Ready for Development"
   в””в”Ђв”Ђ webhook в†’ POST /webhooks/issue-tracker
       в””в”Ђв”Ђ src/routes/issue_tracker.py
           в””в”Ђв”Ђ adapter.parse_webhook() (src/providers/trackers/jira.py)
           в””в”Ђв”Ђ create_state() (src/state/manager.py)
           в””в”Ђв”Ђ threading.Thread(target=run_pipeline_phases, ...)

2. run_pipeline_phases() (src/executor/pipeline.py)
   в”њв”Ђв”Ђ _prepare_repo_for_branch() вЂ” stash, checkout base, pull
   в”њв”Ђв”Ђ adapter.transition_issue() вЂ” Ready for Dev в†’ Development [api mode only]
   в”њв”Ђв”Ђ adapter.read_issue() вЂ” [api mode only, passes ticketData to agent]
   в”‚
   в”њв”Ђв”Ђ Phase 1 вЂ” Analyze
   в”‚   в””в”Ђв”Ђ _run_phase("analyze") with PhaseScope("analyze")
   в”‚       в””в”Ђв”Ђ run_agent() (src/executor/runner.py)
   в”‚           в””в”Ђв”Ђ subprocess.Popen(claude, --allowed-tools=...)
   в”‚               в””в”Ђв”Ђ reads agents/analyze.md
   в”‚               в””в”Ђв”Ђ writes TICKET.md
   в”‚               в””в”Ђв”Ђ outputs __PIPELINE_RESULT__:{"blocked":false}
   в”‚   в””в”Ђв”Ђ pipeline_git.create_feature_branch() вЂ” create branch via API
   в”‚   в””в”Ђв”Ђ pipeline_git.commit_via_api() вЂ” commit TICKET.md via API
   в”‚   в””в”Ђв”Ђ record_phase_end(success)
   в”‚
   в”њв”Ђв”Ђ Phase 2 вЂ” Plan
   в”‚   в””в”Ђв”Ђ _run_phase("plan") with PhaseScope("plan")
   в”‚       в””в”Ђв”Ђ run_agent() (src/executor/runner.py)
   в”‚           в””в”Ђв”Ђ subprocess.Popen(claude, --allowed-tools=...)
   в”‚               в””в”Ђв”Ђ reads agents/plan.md
   в”‚               в””в”Ђв”Ђ writes PLAN.md
   в”‚   в””в”Ђв”Ђ pipeline_git.commit_via_api() вЂ” commit PLAN.md via API
   в”‚   в””в”Ђв”Ђ record_phase_end(success)
   в”‚
   в”њв”Ђв”Ђ Phase 3 вЂ” Implement
   в”‚   в””в”Ђв”Ђ _prepare_repo_for_branch(feature_branch=branch) вЂ” checkout
   в”‚   в””в”Ђв”Ђ _run_phase("implement") with PhaseScope("implement")
   в”‚       в””в”Ђв”Ђ run_agent() (src/executor/runner.py)
   в”‚           в””в”Ђв”Ђ subprocess.Popen(claude, --allowed-tools=...)
   в”‚               в””в”Ђв”Ђ reads agents/implement.md
   в”‚               в””в”Ђв”Ђ writes code, commits locally
   в”‚   в””в”Ђв”Ђ pipeline_git.push_to_remote() вЂ” git push via API
   в”‚   в””в”Ђв”Ђ pipeline_git.create_mr() вЂ” create MR/PR via API
   в”‚
   в””в”Ђв”Ђ transition_state("awaiting-review")
       в””в”Ђв”Ђ adapter.transition_issue() вЂ” Development в†’ Done [api mode only]
       в””в”Ђв”Ђ _try_notify_slack() вЂ” [only if notification enabled]

3. Human reviews the MR on GitLab/GitHub
   в”њв”Ђв”Ђ approves в†’ webhook в†’ POST /webhooks/git (event=approved)
   в”‚   в””в”Ђв”Ђ transition_state("merged")
   в”‚
   в””в”Ђв”Ђ comments в†’ webhook в†’ POST /webhooks/git (event=comment)
       в””в”Ђв”Ђ threading.Thread(target=run_rework_phases, ...)
           в”њв”Ђв”Ђ _run_phase("feedback-parser") вЂ” parse MR comments в†’ FEEDBACK.md
           в”њв”Ђв”Ђ pipeline_git.commit_via_api() вЂ” commit FEEDBACK.md via API
           в”њв”Ђв”Ђ _run_phase("rework") with PhaseScope("rework") вЂ” apply fixes, commit locally
           в”њв”Ђв”Ђ pipeline_git.push_to_remote() вЂ” git push via API
           в””в”Ђв”Ђ transition_state("awaiting-review")
```

## Which Config Controls What?

| Config Field | Affects |
|--------------|---------|
| `repo.mode` | `src/repos/resolver.py` вЂ” how to find the target repo |
| `repo.baseBranch` | `src/repos/resolver.py`, `pipeline.py` вЂ” branch to create features from |
| `issueTracker.type` | `config.py` splits into `platform` + `api_mode`; factory in `issue_tracker.py` picks adapter; `pipeline.py` behavior (api reads ticket, mcp doesn't) |
| `issueTracker.*Status` | Passed to agent via `statuses` input; Python uses in `transition_issue()` calls |
| `gitProvider.type` | Factory in `git_provider.py` picks adapter; `start.sh` generates matching MCP settings.json |
| `cliAdapter.type` | Factory in `cli_adapter.py` picks how to invoke the CLI |
| `notification` (section) | If missing в†’ `get_notification()` returns None в†’ Python never calls Slack в†’ runner adds NO_SLACK_PREAMBLE to agent prompt |
| `pipeline.maxReworkIterations` | Checked in `routes/git_provider.py` вЂ” escalates after N reworks |
| `pipeline.allowCliSkills` | `runner.py` вЂ” if false, prepends SKILLS_DISABLED_PREAMBLE to agent input |
| `pipeline.outputHandlers` | `output_handler.py` registry вЂ” which handlers to load (file, memory, or both) |
| `pipeline.port` | `start.sh` вЂ” uvicorn bind port |
| `pipeline.agentTimeout` | `runner.py` вЂ” agent subprocess timeout |

## Where to Make Common Changes

| I want to... | Edit |
|--------------|------|
| Add a new issue tracker | `src/providers/trackers/newtracker.py` + register in `issue_tracker.py` factory + add to `choices.py` |
| Add a new git provider | `src/providers/git/newgit.py` + register + add MCP server in `mcp_servers/` |
| Add a new AI CLI | `src/providers/cli/newcli.py` + register + add to `choices.py` |
| Change agent behavior | Edit `.md` file in `agents/` |
| Add a new pipeline phase | Add to `VALID_TRANSITIONS` in `state/manager.py`, add phase invocation in `pipeline.py`, create agent prompt file in `agents/` |
| Change what the TUI asks | `installer/setup.py` + `installer/choices.py` |
| Change state file structure | `src/state/manager.py` (and update dashboard if shown) |
| Add an API endpoint | New file in `src/routes/` + register in `server.py` |
| Change dashboard appearance | `console/src/components/*.jsx` + `styles.css` |
| Add a log handler (websocket, webhook, etc.) | New file in `src/providers/output/` extending `OutputHandlerBase` + register in `output_handler.py` |
